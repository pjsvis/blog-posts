# ADVISORY: Requirements Derivation from System Artefacts — Induced vs. Imposed Methodology

**Reference:** RDM-2025-002 | Effective: 2026-05-12 | Review: 2027-05-12
**Authority:** Office of Silo Architecture and Agentic Systems (OSAAS)

---

## PURPOSE

Most system failures originate in a misalignment between stated requirements and the actual state of the system at implementation. The cause is methodological: imposed requirements are generated without examination of the system. This advisory establishes induced methodology as the default approach.

**Imposed:** Generated independently of the system. Applied from above.
**Induced:** Derived from systematic examination of the system. The gap is real; the requirement is pulled from the artefacts.

The distinction matters. The spec and the system are frequently not the same thing.

---

## FINDINGS

### 4.1. The Spec Is Not the System

An imposed requirement is generated by someone who knows what they believe the system should do. This is not the same as knowing what the system actually does. The gap between belief and reality is where technical debt lives.

When the system is built to the requirement, and the requirement does not match the system, the deviation accumulates. Correction is more expensive at implementation than at derivation — this is well-documented. The cost is not in the requirement; it is in the methodology that produced the requirement without reference to the system.

### 4.2. Working Memory Is Not Upgradable

Human personnel engaged in requirements gathering face a cognitive architecture constraint: working memory is finite. Systematic examination of a system requires holding full scope — what exists, what it does, what it should do — simultaneously. This is not a training issue. It is not a motivation issue. It is a ceiling.

Human personnel default to the scope of their current task. The systematic examination required for induced requirements is not being performed because it cannot be performed with the available cognitive resources. This is not apparent to the person. They believe they have examined the system. They have examined their task.

The traditional mitigation — process documentation, architecture reviews, specifications — attempts to externalize scope so it can be examined without being held. This is partially effective. It introduces a new problem: documentation staleness. The documentation describes a system state that existed at the time of writing. The system has changed. The documentation has not.

A stale document is worse than no document. It provides false confidence.

### 4.3. Agents Do Not Have Working Memory Constraints

Agentic systems operating with appropriate orientation read the full project scope, compare it to documented conventions, identify gaps, and draft requirements — without working memory constraints. This is not original design. It is systematic induction: examination executed without the cognitive ceiling that limits human personnel.

The capability is not theoretical. It is operational. The agent holds full scope.

### 4.4. The Human Role Is Judgment, Not Identification

Induced methodology does not eliminate the human contribution. The agent identifies gaps. The human determines whether to address them — given competing demands, resource constraints, and operational context. The agent can state that a gap exists; it cannot state that closing the gap is the correct priority.

Without human triage, the agent surfaces gaps at equal priority regardless of urgency, dependencies, or importance. With triage, the surface is filtered, prioritized, and converted into an actionable backlog.

The agent surfaces. The human decides.

---

## RECOMMENDED PRACTICES

### 5.1. Induced Methodology as Default

Systematic examination precedes requirement generation:

a. Examine the system before generating requirements
b. Identify and record gaps between documented and actual state
c. Formulate requirements in response to identified gaps
d. Review all requirements — induced or imposed — against the examined system state before implementation

### 5.2. Agentic Systems as Examination Infrastructure

Agents performing systematic examination must have appropriate orientation. They must document the examined scope — what was read, compared, and found. Identified gaps route through the brief/debrief process for human triage.

### 5.3. The Registry as Examination Infrastructure

The document registry (`scripts/reg-sync.ts` → `*/INDEX.jsonl`) provides the substrate for systematic examination:

- **Visibility:** Index comparison surfaces gaps between documented and actual state
- **Feedback:** Continuous updates mean gaps surface as the system changes
- **Audit:** Complete history of induced requirements enables retrospective assessment

Gaps in the registry are gaps in examination capability.

### 5.4. When Imposed Requirements Apply

Regulatory requirements; operational constraints that preclude examination; externally issued non-challengeable requirements. All imposed requirements reviewed against system state before implementation. Conflicts documented and escalated.
